# 勤怠管理システム 詳細仕様書 (Project Details)

## 1. データベース設計について
本システムは PostgreSQL を使用し、以下の主要なテーブルで構成されています。用途ごとに分類しています。

### 1.1 データベース設計の基本方針（論理削除）
本システムでは、誤操作によるデータ喪失防止と監査目的のため、原則として**論理削除**（ソフトデリート）を採用しています。
各テーブルには `is_deleted` または `is_active` フラグが設けられており、画面上から「削除」や「取消」操作を行っても物理的なレコード削除（DELETE句）は実行されず、フラグの更新（UPDATE句）が行われます。

### 1.2 ユーザー・マスタ系
- `users`: ユーザー情報（ログイン情報、権限、有給設定、アカウントロック情報、社員番号、雇用形態など）
  - アカウントロック仕様: ログイン失敗3回で30分の一時ロック、5回で永久ロック（管理者による解除が必要）。管理者ユーザー（UserRole.ADMIN）も一般ユーザーと同じ一時ロックの対象となるが、全管理者がロックアウトされる事態を防ぐため永久ロックは管理者には適用せず、一時ロックの延長で対応する。
- `holidays`: 祝日マスタ
- `work_schedule_classes`: 勤務クラス（所属するシフトごとの所定労働時間マスタ）
- `work_schedule_class_breaks`: 勤務クラスごとの休憩時間明細（最大4スロット。`work_schedule_classes` に対する 1:N）
- `event_types`: 勤怠事由マスタ（通常、遅刻、早退、有休、特休、欠勤、土出、休出、テレワークなど15種）
- `system_settings`: システム全体の共通設定値（キー・バリュー形式。システム名、コピーライト、締め日、ファイル名パターンなど）

### 1.3 勤怠・申請系
- `attendance_records`: 日次の出退勤記録（開始・終了時刻、休憩分数、実働時間、残業時間、深夜労働時間、休日出勤フラグなど）
  - **休日出勤時の集計方針**: 休日出勤の実働は `holiday_work_hours` に計上し、`working_hours` は 0.0 とする（総労働と休日労働の二重計上を防ぐ。全入力経路で統一）。
- `attendance_submissions`: 月次勤怠の申請・承認ステータス（対象年月、ステータス、アクション履歴）
- `attendance_correction_requests`: 承認済み勤怠に対する修正申請（申請理由、修正前後の打刻スナップショット、承認ワークフロー用）
- `leave_applications`: 休暇申請（有給休暇、欠勤、特別休暇など。期間種別として終日/午前半休/午後半休を管理）
- `overtime_records`: 残業予定および実績の申請
  - **残業データの正本**: 残業時間の正本は `attendance_records.overtime_hours`（所定終業後の休憩控除込み）。`overtime_records` は打刻・勤怠編集時に同値を同期した派生データであり、残業履歴・給与CSV・勤怠明細で値が一致する。
- `paid_leave_balance`: 各ユーザーの有給休暇の年次残高（年度ごとの付与・利用・繰越の管理）

### 1.4 承認フロー・通知・ログ系
- `attendance_department_approvers` / `attendance_user_approvers`: 部門ごともしくは個別のユーザーごとの承認者（上長）マッピング
- `user_notifications`: ユーザー向けに表示されるダッシュボード通知（勤怠の承認・差し戻し・却下などの通知）
- `admin_announcements`: 管理者から全ユーザーに向けて掲示するお知らせメッセージ
- `audit_logs`: 承認・変更・削除などの重要操作を記録するシステム監査ログ（勤怠修正申請・休暇申請の承認/却下を含む）

### 1.5 初期データ・サンプルデータ
- `src/main/resources/db/sample/V2__Sample_Data.sql`: 開発やテスト環境向けに、以下のテスト用データや初期設定を自動投入するスクリプトです。
  - **ユーザーアカウント**: 管理者（`admin@example.com` / パスワード: `admin123`）、一般ユーザー（`user@example.com` / パスワード: `user123`）、夜勤テストユーザー（`test@example.com` / パスワード: `test123`）の3名。
  - **勤務クラス（シフト）マスタ**: 「標準勤務」「早番」「遅番」「時短勤務」「夜勤」の5パターンの所定時間設定。
  - **システム共通設定**: 勤怠の締め日設定、CSVファイル名パターン、有給付与タイミングやリマインダー通知実行日など。
  - **各種トランザクションデータ**: 出退勤履歴、休暇申請、有給休暇の初期残高とお知らせメッセージ。

## 2. 画面毎に実装されている機能について

画面（Thymeleafテンプレート）は、共通レイアウト (`layout/base.html`)、一般ユーザー向け(`user/`)、管理者向け(`admin/`)、その他に分かれています。
スタイルは Tailwind CSS で構成されており、ビルド時に `tailwind.css` が生成されます。
ロール別のアクセス制御は Spring Security および `@PreAuthorize` アノテーションにより実施されています。

#### ⑥ 休暇申請新規登録フォーム
- **URL**: `/leave/create`
- **テンプレート**: `user/leave-create.html`
- **遷移経路**: 休暇申請一覧の「新規休暇申請」ボタン
- **入力機能**:
  - **休暇設定**: 開始日・終了日のカレンダー入力
  - **休暇タイプ**: プルダウン選択（有給休暇、無給休暇、生理休暇、特別休暇、欠勤など）
  - **期間種別**: 終日／午前半休／午後半休のラジオボタン
  - **申請理由**: 理由のテキスト入力
  - **送信**: 「申請する」ボタン（POST `/leave/create`）。※`/leave/apply`・`/leave/applyPaid` は月次勤怠画面（③）の日別ワンクリック申請が使用する別エンドポイント
    - **【重要・自動承認仕様】** 休暇申請は、送信と同時にシステムで**自動的に即時承認（APPROVED）**される仕様です（承認ワークフローを経由しません）。作成と即時承認は単一トランザクションで実行され、承認時に有給残高不足などの例外が発生した場合は作成も含めてロールバックされます（PENDING 申請は残留しません）。
    - **【有給消化日数の算定基準】** 有給休暇（PAID_LEAVE）の消化日数は、申請期間のうち**所定労働日のみ**を計上します。土曜・日曜および `holidays` マスタに登録された祝日は消化日数に含めません（暦日ではなく所定労働日ベース）。午前半休・午後半休は単日申請のみ許可され、0.5日として計上します。
- **出力機能**: なし（登録後は一覧画面へリダイレクト）

### 2.2 承認者（マネージャー）向け画面

> 一般ユーザーロールを持ちながら、配下ユーザーの承認者として設定されている場合にアクセス可能。

#### ① 勤怠承認待ち一覧
- **URL**: `/attendance/approval`
- **テンプレート**: `user/attendance-approval.html`
- **遷移経路**: ダッシュボードの「勤怠承認待ち」件数リンク、またはサイドバーリンク
- **入力機能**:
  - **個別承認**: 対象の申請レコードに対しコメントを付与して承認（POST `/attendance/approval/{submissionId}/approve`。申請者に承認通知を送信）
  - **差戻し**: 対象の申請レコードに対しコメントを付与して差し戻し（POST `/attendance/approval/{submissionId}/return`。申請者に差し戻し通知を送信）
- **出力機能**:
  - 承認待ち従業員の一覧（氏名、対象月、提出日時、ステータス）

#### ② 修正申請承認待ち一覧
- **URL**: `/attendance/approval/corrections`
- **テンプレート**: `user/correction-approval.html`
- **遷移経路**: ダッシュボードの「修正申請承認待ち」件数リンク、またはサイドバーリンク
- **入力機能**:
  - **個別承認**: 差分確認後、コメントを付与して承認（POST `/attendance/approval/corrections/{requestId}/approve`。申請者に承認通知を送信し、勤怠レコードを自動上書き更新する）
  - **却下**: コメントを付与して却下（POST `/attendance/approval/corrections/{requestId}/reject`。申請者に却下通知を送信）
- **出力機能**:
  - 修正申請中の従業員一覧、および修正前・修正後のデータ差分比較表示

#### ③ 従業員勤怠詳細画面
- **URL**: `/attendance/approval/{userId}/detail`
- **テンプレート**: `admin/user-attendance-detail.html`
- **遷移経路**: 勤怠承認待ち一覧や勤怠管理画面の「詳細」リンク
- **入力機能**:
  - **勤怠記録の一括保存**: 指定した月の開始・終了時間、備考、イベントタイプを一括で上書き保存する（POST `/attendance/approval/{userId}/detail/saveAll`）
    - **【重要・権限制限】** この一括保存機能は**管理者（ADMIN）ロールのみ実行可能**です。一般の承認者（マネージャー）は実行できません（アクセス拒否となります）。
- **出力機能**:
  - 指定従業員の月次勤怠明細（日別記録、実働・残業・深夜・休日出勤時間、備考、ステータス）
  - 36協定状態（安全/注意/超過）
  - 有休消化状況
  - ロールや遷移元（`from=admin` など）に応じてダッシュボード、勤怠管理、承認待ち一覧などへ動的に変化する戻り先リンク（`backUrl`）
- **閲覧権限の制限**:
  - 一般の承認者は、**自身と同じ勤務クラス（所属部署）のユーザーのみ**詳細を閲覧可能です。異なるクラスのユーザー詳細にアクセスした場合はアクセスが拒否されます。管理者は制限なく全ユーザーの詳細を閲覧できます。

  - **パスワード初期化**: 「一時パスワードを発行」ボタン（POST `/admin/users/{userId}/reset-password`。一時パスワードを発行し次回ログイン時に強制変更させる）

- **出力機能**:
  - 従業員個人の登録情報、割り当てられている承認者、および現在の有給付与パラメータ設定値
